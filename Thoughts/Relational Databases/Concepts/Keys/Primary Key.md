# Primary Key

A **Primary Key (PK)** is a special type of key in a relational database table that uniquely identifies each [[Rows|row]] in that table. It is a fundamental concept in relational database design, crucial for maintaining data integrity and establishing [[One-to-Many|relationships]] between tables.

## Characteristics of a Primary Key

1.  **Uniqueness**: Each value in the primary key column (or combination of columns) must be unique. No two rows can have the same primary key value. This ensures that every row can be unambiguously identified.
2.  **Non-Nullability**: A primary key cannot contain `NULL` values. Every row must have a primary key value. This is often referred to as "Entity Integrity."
3.  **Stability**: Ideally, primary key values should not change over time. Changes to primary keys can have cascading effects on related tables, making database maintenance more complex.
4.  **Minimality**: A primary key should consist of the minimum number of columns necessary to ensure uniqueness. If a single column is sufficient, then multiple columns should not be used.

## Purpose of a Primary Key

*   **Unique Identification**: Provides a guaranteed way to locate and refer to any specific row in a table.
*   **Data Integrity**: Enforces entity integrity, preventing duplicate records and ensuring that every record has an identifier.
*   **Relationship Establishment**: Serves as the "parent" side of a relationship with a [[Foreign Key]] in another table, linking related data across different tables. This is how data from multiple tables can be combined and queried efficiently.
*   **Indexing**: Database systems typically create an index on the primary key automatically, which speeds up data retrieval and query performance.

## Example

Consider two tables: `Students` and `Courses`.

### Students Table

| StudentID (PK) | FirstName | LastName |
| :------------- | :-------- | :------- |
| 101            | Alice     | Smith    |
| 102            | Bob       | Johnson  |
| 103            | Charlie   | Brown    |

In the `Students` table, `StudentID` is the Primary Key. Each student has a unique `StudentID`, and no two students can share the same ID. Also, `StudentID` cannot be `NULL`.

### Courses Table (Illustrative, showing relationship)

| CourseID (PK) | CourseName      | StudentID (FK) |
| :------------ | :-------------- | :------------- |
| 501           | Database Systems| 101            |
| 502           | Web Development | 102            |
| 503           | Algorithms      | 101            |

Here, `CourseID` is the Primary Key for the `Courses` table. The `StudentID` column in the `Courses` table would be a [[Foreign Key]] referencing the `StudentID` Primary Key in the `Students` table, linking courses to the students enrolled in them.

## Choosing a Primary Key

When designing a database, it's important to choose an appropriate primary key. Common choices include:

*   **Auto-incrementing Integers**: Often generated by the database system (e.g., `IDENTITY` in SQL Server, `AUTO_INCREMENT` in MySQL, `SERIAL` in PostgreSQL). These are simple, efficient, and never change.
*   **Unique Application-generated IDs**: Such as UUIDs (Universally Unique Identifiers).
*   **Natural Keys**: Existing unique attributes in the data, like a Social Security Number or ISBN. However, natural keys can sometimes change or be subject to business rule alterations, making auto-incrementing surrogate keys generally preferred.

## Composite Primary Key

Sometimes, a single column is not sufficient to uniquely identify a row. In such cases, a **composite primary key** (or compound primary key) is used, which consists of two or more columns whose values, when combined, uniquely identify each row.

*Example*: In a table tracking enrollment, (`StudentID`, `CourseID`) might form a composite primary key, as a student can enroll in multiple courses, and a course can have multiple students, but a specific student can only be enrolled in a specific course once.

Understanding primary keys is essential for designing robust, efficient, and reliable relational databases.
